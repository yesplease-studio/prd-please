# Skill: prd-validator

**Purpose:** Validate that implementation work satisfies the requirements, acceptance criteria, and guardrails defined in the Strategic PRD. This skill checks compliance — it does not build or modify code.

---

## When to Use

- After a task is implemented, before marking it complete.
- During code review as an automated compliance check.
- When the human or another agent requests a validation pass.
- As a pre-merge gate in CI/CD pipelines (when tooling supports it).

## Core Principle

Validation is a **checklist operation**, not a reasoning task. The Strategic PRD defines what must be true; this skill checks whether it is true. This makes it a strong candidate for sub-agent delegation to cheaper models.

---

## Validation Workflow

### Step 1: Load Task Context

Load the task context generated by `prd-taskmaster` for the task being validated. This includes:

- Referenced requirement IDs and their acceptance criteria.
- Applicable guardrails (do's and don'ts).
- Relevant learnings.

If no task context exists (e.g. validating ad-hoc work), fall back to loading the rules summary YAML for the relevant PRD. If that doesn't exist either, load Sections 5, 6, and 8 of the Strategic PRD directly.

### Step 2: Gather Implementation Context

Identify what was built:

- Changed files (from git diff, staged changes, or provided file list).
- The scope of the change (which components, services, or modules were touched).

### Step 3: Launch Domain Sub-Agents

For each domain represented in the task's requirements, launch a sub-agent to check compliance. Run all sub-agents **in parallel**.

**CRITICAL: Set `model: "haiku"` on every sub-agent.** This is checklist verification, not complex reasoning. Save frontier model tokens for the orchestration layer.

**Always launch a Guardrails sub-agent** (checks Section 6 compliance across all changed files).

**Launch domain-specific sub-agents** only for domains with requirements in the task.

#### Sub-Agent: Guardrails Check

```
You are checking code changes for compliance with project guardrails.

## Changed Files
{CHANGED_FILES_LIST}

## Guardrails to Check
{GUARDRAILS_FROM_TASK_CONTEXT}

## Instructions
1. Read each changed file.
2. For each DO guardrail: verify the code follows it. Flag if not.
3. For each DON'T guardrail: verify the code does not violate it. Flag if it does.
4. Only check guardrails that are relevant to the changed files. Skip guardrails about domains not touched by this change.

## Output Format (STRICT — keep under 20 lines)
CHECK: Guardrails
STATUS: PASS | FAIL | PASS_WITH_WARNINGS
VIOLATIONS: None | <one per line: guardrail text → file:line → issue>
WARNINGS: None | <one per line>
```

#### Sub-Agent: Domain Requirements Check

```
You are checking code changes for compliance with {DOMAIN} requirements.

## Changed Files
{CHANGED_FILES_FOR_DOMAIN}

## Requirements to Verify
{REQUIREMENTS_WITH_ACCEPTANCE_CRITERIA}

## Relevant Learnings
{LEARNING_ENTRIES_FOR_DOMAIN}

## Instructions
1. Read each changed file.
2. For each requirement (by ID): check whether the acceptance criteria are satisfied.
3. For each learning entry: check whether the learned rule is being followed.
4. Only check items relevant to the changed files. Don't flag requirements that aren't addressed by this task.

## Output Format (STRICT — keep under 25 lines)
CHECK: {DOMAIN}
STATUS: PASS | FAIL | PASS_WITH_WARNINGS
REQUIREMENTS_MET: <comma-separated list of met requirement IDs>
REQUIREMENTS_UNMET: None | <one per line: REQ-ID → issue → suggestion>
LEARNING_COMPLIANCE: PASS | <one per line: L-XXX → violation>
WARNINGS: None | <one per line>
```

### Step 4: Aggregate Results

After all sub-agents return, produce a concise compliance report:

```markdown
## Validation: APPROVED | BLOCKED

**Task:** T-001 — Implement user authentication API
**PRD:** PRD-001

| Check | Status | Issues |
|-------|--------|--------|
| Guardrails | PASS | — |
| Tech | PASS | — |
| Product | FAIL | FR-03: password reset flow missing |

**Violations (1):**
- FR-03 (must): Password reset endpoint not implemented. Acceptance criterion "POST /auth/reset sends email within 30s" not met. **Fix:** Add /auth/reset endpoint with email service integration.

**Warnings (0):** None

**Requirements coverage:** 5/6 met (83%)
```

### Decision Framework

| Condition | Verdict |
|-----------|---------|
| All sub-agents PASS or PASS_WITH_WARNINGS | **APPROVED** |
| Any `must` requirement unmet | **BLOCKED** — provide specific fix suggestions |
| Any `should` requirement unmet | **APPROVED_WITH_WARNINGS** — note the gap, suggest follow-up |
| Any `may` requirement unmet | **APPROVED** — note as optional improvement |
| Learning violation detected | **BLOCKED** — the team already learned this lesson; it must be fixed |

**Learning violations are treated as seriously as `must` violations.** The entire point of the learning mechanism is to prevent repeated mistakes. If a learning entry says "don't do X" and the implementation does X, that's a hard block.

---

## Escalation

The validator escalates to a human when:

- A `must` requirement appears impossible to satisfy given the current architecture (may indicate a PRD issue, not just an implementation gap).
- Two requirements conflict in practice (e.g. a performance requirement conflicts with a security requirement).
- The implementation intentionally deviates from a `must` requirement and includes documentation explaining why.

In escalation cases, the validator does **not** issue a verdict. It presents the situation and asks for a human decision.

---

## What This Skill Does NOT Do

- **Does not build or modify code.** It only checks compliance.
- **Does not modify the PRD.** If it discovers a gap in the PRD (e.g. a requirement that's untestable), it flags it for `prd-learner`.
- **Does not run tests.** It checks against PRD requirements, not test suites. Test execution is a separate concern.
- **Does not make architectural decisions.** If the implementation takes a different approach than the PRD suggests but still meets all requirements and acceptance criteria, that's a PASS.

## Beyond PRD Compliance

This skill checks whether implementation satisfies the requirements, acceptance criteria, and guardrails in the Strategic PRD. It is **not** a substitute for a full QA process. Two areas that commonly come up:

### UI and visual testing

PRD compliance for design requirements (e.g. `DES-01: Chart builder is usable by someone who has never written SQL`) can be partially verified by checking that the relevant components exist and follow spec — but visual correctness, layout fidelity, and interaction quality require separate tools.

**Recommended approach:**
- For visual regression testing, use a tool like Playwright (screenshot diffs) or Chromatic (Storybook-based).
- For accessibility validation (WCAG compliance), use axe-core or Lighthouse as part of CI.
- Claude Code with browser access can run basic interaction walkthroughs when given a running URL.

When a design requirement has an acceptance criterion that cannot be verified programmatically (e.g. "3/5 non-technical participants can complete the flow in under 3 minutes"), this skill will flag it as **cannot verify automatically** rather than marking it as PASS or FAIL. Human or usability testing is required for those criteria.

### Automated test suite execution

This skill reads code to check PRD compliance — it does not run test suites. Test execution (unit, integration, e2e) is a separate step and should happen before or alongside validation.

**Recommended chain:**
```
Build
  → Run test suite (pytest / jest / playwright / etc.)
  → prd-validator (compliance against PRD requirements)
  → Ship if both pass
```

If a test suite exists and is passing but the PRD validator still flags violations, it means the tests don't cover all PRD acceptance criteria. That gap is itself a finding worth capturing via `prd-learner`.

### When to extend the validation chain

If your PRD has requirements that need specialized verification beyond code reading (load testing, security scanning, accessibility audits), add those as explicit steps in the workflow YAML before or after `prd-validator`. The validator's compliance report is designed to be one input in a broader pre-ship checklist, not the entire checklist.

## Token Optimization

- **Sub-agents on Haiku.** The orchestrator (this skill) runs on a frontier model to handle parsing, routing, and aggregation. The actual file-by-file checking runs on the cheapest model that can follow a checklist.
- **Strict output limits.** Sub-agents are constrained to 20-25 lines of output. This prevents verbose analysis that wastes tokens in the aggregation step.
- **Load task context, not full PRD.** The task context is the minimal set of requirements and guardrails needed. The full PRD is only loaded if no task context exists.
- **Skip irrelevant domains.** If the task only touches backend code, don't launch a Design sub-agent even if the PRD has design requirements.
