# PRD System

A system for writing, structuring, and evolving product requirements so that AI agents can reliably scope, build, validate, and learn from them. Humans provide vision, context, and approval. AI handles authoring, execution, and continuous improvement.

---

## 1. Core Concepts

### 1.1 What the System Does

The PRD System is the translation layer between human intent and AI execution. It takes messy inputs — call transcripts, client docs, sketches, ideas — and produces structured artifacts that AI agents can act on immediately and repeatedly.

It solves three problems:

1. **Distillation.** Converts unstructured human context into a precise, structured source of truth (the Strategic PRD).
2. **Operationalization.** Derives executable tasks from the source of truth so building can begin within hours, not days.
3. **Continuous learning.** Feeds implementation experience back into the source of truth so the same mistakes are never repeated.

### 1.2 The Three Layers

Every product engagement produces artifacts at three layers:

| Layer | Artifact | Audience | Lifespan | Authored by |
| --- | --- | --- | --- | --- |
| **Strategic** | The Strategic PRD | Internal team, AI agents | Long-lived, updated over time | `prd-author` skill |
| **Execution** | Tasks + Client View | Builders (AI/human) + Client | Short-lived, per-task | `prd-taskmaster` skill |
| **Learning** | Amendments to the Strategic PRD | Future agents and sessions | Permanent | `prd-learner` skill |

The Strategic PRD is the only manually reviewed artifact. Everything at the Execution and Learning layers is **derived** from it — generated by skills, not authored separately. This keeps the system consistent: one source of truth, multiple projections.

### 1.3 The Five Skills

The system's operations are decomposed into five discrete skills:

| Skill | Role | When Used | Model Guidance |
| --- | --- | --- | --- |
| `prd-discovery` | Structured pre-authoring interview that surfaces what is known, inferred, and unknown across all domains — produces a Discovery Brief | Before authoring any new PRD where input material is thin or one-sided | Frontier model (conversational reasoning, synthesis) |
| `prd-author` | Create and edit Strategic PRDs from human input or a Discovery Brief | Start of engagement; scope changes; major pivots | Frontier model (complex reasoning) |
| `prd-taskmaster` | Derive executable tasks and client-facing views from a Strategic PRD | Every sprint/cycle; incoming client requests | Frontier for decomposition; sub-agents on cheaper models for formatting |
| `prd-validator` | Check implementation against PRD requirements and rules | During and after building | Sub-agents on cheaper models for checklist verification |
| `prd-learner` | Capture implementation learnings and amend the Strategic PRD | After building; after incidents; after validation failures | Frontier model (judgment required) |

Skills are invoked independently and compose into workflows. They never overlap in responsibility: `prd-discovery` interviews and synthesizes but never writes the PRD; only `prd-author` writes the Strategic PRD (with `prd-learner` proposing amendments through it); only `prd-taskmaster` generates tasks, etc.

## 2. The Strategic PRD

### 2.1 What It Is

A single markdown document per product or major module. It is:

- The **authoritative source of truth** for what to build and why.
- **Structured for AI parsing** with consistent sections, frontmatter metadata, and domain-tagged requirements.
- **A living document** that accumulates learnings and amendments over time.
- **Authored primarily by AI** using human-provided context (transcripts, docs, discussions). Humans review and approve; they do not write.

### 2.2 Frontmatter

```yaml
---
id: PRD-001
title: "Feature Name – Phase"
product: "Product Name"
area: core-feature
status: draft                   # draft | active | deprecated
phase: R1                       # roadmap phase (R1, R2, etc.)
priority: P0                    # P0 (must-have) | P1 (important) | P2 (nice-to-have)
owner: owner.name
contributors:
  - name.one
  - name.two
depends_on:
  - PRD-002
labels:
  - ai-native
links:
  design: https://...
  transcript: https://...
  repo: https://...
created_at: 2026-03-01
updated_at: 2026-03-01
---
```

**Key fields for agents:**

- `depends_on` — Load these PRDs as additional context before working on this one.
- `status` — Only `active` PRDs govern implementation. `draft` PRDs are under review. `deprecated` PRDs are historical.
- `phase` — Determines what's in scope for the current build cycle.
- `priority` — P0 requirements are non-negotiable; P1/P2 allow trade-offs.

### 2.3 Body Structure (8 Sections)

```
## 1. Context & Problem
## 2. Goals, Non-goals & Success Metrics
## 3. Users & Scenarios
## 4. Solution Overview
## 5. Requirements by Domain
## 6. Guardrails
## 7. Risks & Open Questions
## 8. Learnings
```

**Section 1: Context & Problem**

Background, current state, and a concise articulation of what is wrong or missing. Include relevant history, existing workflows, and pain points. This section answers: *why does this PRD exist?*

**Section 2: Goals, Non-goals & Success Metrics**

- **Goals** — What this product/feature achieves, scoped by phase (e.g. R1, R2). Each goal should be concrete and measurable where possible.
- **Non-goals** — What is explicitly out of scope. This is as important as goals — it prevents scope creep and gives agents clear boundaries.
- **Success metrics** — How we know this worked. Quantitative where possible, qualitative where necessary.

**Section 3: Users & Scenarios**

- **User roles** — Primary and secondary users with brief descriptions.
- **User stories** — Atomic stories with IDs, role, phase tag, and priority. Format: `US-01 (Role, R1, MUST): As a [role], I want [action] so that [outcome].`
- **Key scenarios** — 2-3 narrative walkthroughs showing how users interact with the product end-to-end.
- **Buyer or stakeholder journey** (when applicable) — Maps the lifecycle beyond product usage. Which variant applies depends on context:
  - **External products** — Buyer journey stages: *Unaware → Problem-aware → Evaluating → Onboarding → Active → Expanding → Graduating/Churning*. For each relevant stage: who is involved (buyer, champion, end user), what they need, how the product shows up, and what triggers progression to the next stage. This captures the commercial lifecycle that shapes GTM requirements in Section 5.
  - **Internal tools** — Stakeholder journey stages: *Need identified → Build-vs-buy evaluated → Sponsored → Rolled out → Adopted → Maintained*. For each relevant stage: who owns the decision, what tools the product replaces, how value is justified, and what the long-term maintenance model looks like. This captures the organizational lifecycle that shapes internal GTM requirements in Section 5.
  - Not every PRD needs this. Skip it for purely technical modules, infrastructure work, or features with no commercial or stakeholder dimension.

**Section 4: Solution Overview**

High-level description of the solution approach. Architecture at a glance — components, services, integrations. Enough for a builder to understand the shape of what they're building before diving into requirements.

**Section 5: Requirements by Domain**

This is the core of the PRD. Requirements are organized by **domain subsections**. Each requirement has:

- **Requirement ID** (e.g. `PR-01`, `TECH-01`, `DATA-01`)
- **Severity** — `must` | `should` | `may`
- **Description** — What the requirement demands.
- **Acceptance criteria** — Testable conditions for this specific requirement.
- **Phase** — Which release/phase this applies to.

**Domain subsections** (include only those relevant to the PRD):

| Domain | Covers | Example Requirements |
| --- | --- | --- |
| **Product** | Core functionality, user-facing behavior, UX flows | "User can filter dashboard by date range" |
| **Tech** | Architecture, infrastructure, performance, security | "API response time < 200ms at p95" |
| **Data** | Data models, pipelines, privacy, analytics | "PII must be encrypted at rest" |
| **Design** | UI/UX patterns, accessibility, visual standards | "All interactive elements meet WCAG AA" |
| **AI** | Agent behavior, prompts, guardrails, evaluation | "Agent must not fabricate data points" |
| **GTM** | **External:** Market & ICP, pricing strategy & packaging, buyer journey requirements, sales motion (self-serve / PLG / sales-led), positioning & messaging. **Internal:** Stakeholders & owners, tools replaced & migration path, build-vs-buy justification, maintenance model, integration surface with existing tools. | "Free tier limited to 3 projects" / "Replaces manual Sheets workflow for ops team" |
| **General** | Cross-cutting conventions, documentation, processes | "All API endpoints must have OpenAPI specs" |

**The domain structure serves as a forcing function.** Even if the initial input material only discusses product features, the `prd-author` skill should prompt the human to consider tech constraints, data implications, design standards, and commercial considerations.

**GTM is context-dependent.** The `prd-author` interview determines whether the product is external (sold to customers) or internal (built for the team), and applies the corresponding GTM sub-context. GTM is optional — purely technical modules, infrastructure work, and features without a commercial or stakeholder dimension can omit it entirely.

**Section 6: Guardrails**

Explicit do's and don'ts that apply across the entire PRD. These are the rules that agents must internalize before building.

- **Do's** — Prescriptive behaviors. Prefix each with **DO**. Be specific: name patterns, tools, APIs, conventions.
- **Don'ts** — Prohibited behaviors. Prefix each with **DON'T**. Name specific anti-patterns, not vague warnings.
- **AI-specific guardrails** (when applicable) — Constraints on AI agent behavior.

Guardrails must be **objectively verifiable**. "Write clean code" is not a guardrail. "All functions must have JSDoc comments" is.

**Section 7: Risks & Open Questions**

- **Risks** — Known risks with severity assessment and mitigation strategy. Each risk should have an owner.
- **Open questions** — Unresolved decisions with IDs (e.g. `OQ-01`), description, owner, and target resolution date.
- **References** — Links to related PRDs, design docs, transcripts, recordings, external resources.

Open questions must be resolved before their associated requirements can move to implementation.

**Section 8: Learnings**

This section starts empty and **accumulates over time**. It is the system's memory — the mechanism that prevents repeated mistakes.

Each learning entry follows this structure:

```
### L-001: [Short title]
- Date: YYYY-MM-DD
- Domain: [which domain this relates to]
- Related requirements: [requirement IDs affected]
- What happened: [Description of the incident or discovery]
- Root cause: [Why it happened]
- Amendment made: [What changed in the PRD as a result]
- Status: resolved | monitoring
```

When a learning results in a change to the PRD, the change is made **inline in the relevant section** AND logged here with context. This way:

- An agent loading sections 5-6 for a task gets the **current rules** (including all amendments).
- An agent that needs to understand **why** a rule exists can check the Learnings section.

## 3. Derived Artifacts

These artifacts are **generated from the Strategic PRD by skills**, never authored manually.

### 3.1 Rules Summary (YAML)

A machine-optimized extraction of all requirements and guardrails from the Strategic PRD. Generated by `prd-taskmaster` when needed. Contains only requirement IDs, severities, descriptions, and acceptance criteria — the minimal payload for coding agents.

### 3.2 Execution View (Client-Facing)

A simplified, non-technical summary of what's being built and why. Generated by `prd-taskmaster` for the client-facing platform (e.g. kanban board). For each task or feature slice, it includes: what is being built (plain language), why it matters, what the client will see when it's done, and current status.

### 3.3 Task Context (Agent-Facing)

The minimal payload a coding agent loads when picking up a specific task. Includes: task description, referenced requirement IDs and acceptance criteria, applicable guardrails, relevant learnings, and dependencies. This is the key token optimization: instead of loading an entire PRD for every task, the agent loads a focused slice.

## 4. ID Conventions

### 4.1 PRD IDs

PRDs use a simple sequential ID: `PRD-001`, `PRD-002`, etc. For cross-cutting vision documents, use the `GEN-` prefix: `GEN-001`.

### 4.2 Requirement IDs

Requirements within a PRD are prefixed by their domain:

| Domain | Prefix | Example |
| --- | --- | --- |
| Product | `PR-` | `PR-01`, `PR-02` |
| Tech | `TECH-` | `TECH-01`, `TECH-02` |
| Data | `DATA-` | `DATA-01`, `DATA-02` |
| Design | `DES-` | `DES-01`, `DES-02` |
| AI | `AI-` | `AI-01`, `AI-02` |
| GTM | `GTM-` | `GTM-01`, `GTM-02` |
| General | `GEN-` | `GEN-01`, `GEN-02` |

When referencing across PRDs, use the full form: `PRD-001:PR-01`.

### 4.3 Other IDs

- **User stories:** `US-01`, `US-02`, etc. (scoped to PRD)
- **Open questions:** `OQ-01`, `OQ-02`, etc. (scoped to PRD)
- **Learnings:** `L-001`, `L-002`, etc. (scoped to PRD, sequential over time)

## 5. When to Use (and Not Use) This System

The PRD system is designed for work where **you are authoring the requirements** — deciding what to build, how it should behave, and what constraints apply. It pays for itself when the build spans weeks, involves collaborators, or will evolve through multiple phases.

It is the wrong tool when the requirements are externally fixed, the timeline is hours not weeks, or the work is exploratory and likely to be thrown away.

### Use the PRD system when:

| Context | Why it fits |
|---------|-------------|
| Client product engagement (weeks/months) | Full 8-section structure earns its weight. Domain forcing catches blind spots. Learnings compound over time. |
| Internal tool with multiple users or collaborators | Shared source of truth prevents drift. Task derivation keeps builders aligned. |
| Extending an existing product (R2, R3) | Edit mode, learnings section, and phase-gating are purpose-built for this. |
| Ambiguous scope that needs to be pinned down | The interview and outline phases force clarity before building starts. |

### Do not use the PRD system when:

| Context | Why it doesn't fit | Use instead |
|---------|---------------------|-------------|
| Hackathon, competition, or time-boxed sprint (hours) | The 8-section structure adds ceremony that doesn't pay back in a short window. Requirements don't need to be "lived in." | A brief: context, approach, build order. One page. |
| Exploratory prototype or spike | You don't know enough for requirements yet. Writing them creates false confidence. | A hypothesis and a timebox. Write the PRD after the spike confirms the direction. |
| Integration work with a fixed external spec | The requirements are given, not authored. The PRD's value is in deciding *what* — when the what is defined externally, you're documenting, not deciding. | A technical brief or implementation plan that references the external spec directly. |
| One-off script or automation | No users, no phases, no learning loop. The overhead exceeds the artifact's useful life. | Inline comments and a README if it'll be reused. |

### How to decide

Ask two questions:

1. **Am I authoring the requirements, or are they given to me?** If given → skip the PRD.
2. **Will this work be revisited, extended, or maintained?** If no → the learning loop and phase structure won't pay off.

If both answers point away from a PRD, use a lighter format. The research and context-gathering that `prd-discovery` and `prd-author` do (fetching docs, interviewing, forcing domain completeness) is still valuable — but the output should match the work's weight.

## 6. Severity Semantics

| Severity | Meaning | Agent Behavior |
| --- | --- | --- |
| `must` | Non-negotiable. Hard requirement. | Treat as a blocker. Never skip. Surface conflicts immediately. |
| `should` | Strong guidance. Deviations need reasoning. | Follow unless there's a documented trade-off. Explain deviations. |
| `may` | Optional. Nice-to-have. | Implement if straightforward. Skip if it creates complexity. |

## 7. Workflows

Workflows are defined as YAML files in `workflows/product/` and executed by pointing an agent at the file. Each workflow chains skills sequentially, passes outputs between steps, and includes human approval gates.

### 7.1 Starting a New Engagement

**Workflow file:** `workflows/product/prd-new-engagement.yaml`

The full engagement workflow runs discovery first (optional but recommended), then authors the PRD, gates on human approval, and generates the task backlog.

```
prd-discovery skill (optional)
[Structured interview → Discovery Brief]
        ↓
   prd-author skill
   [Ingests Discovery Brief + any source material]
   [Conversational, phased, human-in-the-loop]
        ↓
   Strategic PRD (draft)
        ↓
   Human review & approval
        ↓
   Strategic PRD (active)
        ↓
   prd-taskmaster skill
        ↓
   Executable tasks + Client view
        ↓
   Building begins
```

### 7.2 Handling an Incoming Client Request

No dedicated workflow file — invoke `prd-taskmaster` directly.

```
Client request (kanban ticket, message, etc.)
        ↓
   prd-taskmaster skill
   [Maps request against existing PRD]
        ↓
   Decision:
   ├── Fits existing scope → Generate task with context
   ├── Extends scope → Flag for prd-author to amend PRD
   └── Out of scope → Flag for human decision
```

### 7.3 Post-Build: Validate, Learn, and Amend

**Workflow file:** `workflows/product/prd-post-build.yaml`

```
prd-validator skill
[Check implementation against requirements]
        ↓
   Pass → Ship
   Fail → Fix violations, re-validate
        ↓
   prd-learner skill
   [Proposes amendments from implementation experience]
        ↓
   Human approval
        ↓
   prd-author applies amendments to Strategic PRD
        ↓
   prd-taskmaster regenerates affected task contexts
```

## 8. Design Principles

1. **One source of truth.** The Strategic PRD is the single authoritative artifact. Everything else is derived from it.
2. **Humans decide, AI executes.** Humans provide vision, context, and approval. AI handles authoring, decomposition, validation, and learning capture.
3. **Domains force completeness.** The domain structure exists to ensure comprehensive product thinking — product, tech, data, design, AI, GTM, general. `prd-discovery` applies this forcing function before authoring begins; `prd-author` applies it again during the interview phase.
4. **Every mistake becomes a permanent rule.** The learning mechanism ensures that implementation failures get codified back into the PRD.
5. **Token efficiency through derived artifacts.** Agents building features load focused task contexts, not full PRDs.
6. **Platform agnostic.** The system is independent of any specific tool, platform, or storage system.
